# 2026-03-04 habiDue 프로젝트 기술 고도화 상세 리포트

## 1. 프론트엔드 UI/UX 혁신 및 성능 최적화

### 1.1 통합 서클 필터 바 (Integrated Circle Filter Bar) 도입
- **변경 사항**: 기존의 텍스트 배지형 필터와 기관별 필터를 인스타그램 스토리 스타일의 **동그라미 서클 UI**로 대통합.
- **기술적 해결**:
    - **다중 선택 로직**: 기관(OR) + 상태(OR) 필터를 조합하되, 기관과 상태 사이는 AND 조건으로 작동하도록 서버 쿼리 연동.
    - **서버 사이드 필터링 전향**: 기존에 현재 페이지(10~15건) 내에서만 작동하던 클라이언트 필터링을 제거하고, 필터 클릭 시마다 DB 전체 데이터를 조회하도록 수정하여 데이터 누락 원천 차단.
- **UI 튜닝**: 
    - **다크모드 가독성**: 다크모드에서 배경색과 글자색이 겹쳐 보이지 않던 정렬 셀렉트 박스(`Select Box`)를 테마 변수(`var(--card-bg)`)와 `!important` 속성을 활용해 가독성 확보.
    - **모바일 최적화**: `NoticeCard`의 여백을 6px로 정밀 조정하여 정보 밀도를 높이고, 컨텍스트 메뉴(...)를 상단 방향으로 팝업되도록 수정하여 엄지손가락 조작 편의성 극대화.

## 2. 전략적 태그 시스템 (Strategic Tagging System) 전면 개편

### 2.1 단일 진실 공급원 (Single Source of Truth) 구축
- **NoticeKeyword Enum 대통합**: 사용자 요청에 따라 약 150여 개의 부동산 전문 키워드를 카테고리별로 단일 파일(`NoticeKeyword.java`)에서 관리하도록 설계.
    - **행정/절차**: 정정, 안내, 당첨자발표, 서류제출 등
    - **주거유형**: 행복주택, 매입임대, 든든전세, 신혼희망타운 등
    - **공급대상**: 신혼부부, 신생아, 1인가구, 생애최초, 줍줍 등
    - **금융/입지**: 역세권, 5년무이자, 보증금완화, 전매제한 등
    - **상세지역**: 고양삼송, 성남판교 등 30여 개 주요 거점 지역

### 2.2 지능형 태그 엔진 (TagService) 고도화
- **다양성 정책 반영**: 동일한 키워드가 시스템 필터용(`SYSTEM`)과 일반 정보용(`TYPE`, `SPECIAL`)으로 공존할 수 있도록 허용.
- **데이터 정규화**: "서울특별시" -> "서울" 등 지역 명칭을 표준화하여 저장함으로써 검색 효율성 증대.
- **자동 상태 갱신**: 수집기가 돌 때마다 `deadline`을 실시간 계산하여 `접수중` 태그를 `마감`으로 자동 교체하는 동적 태깅 시스템 구현.
- **정밀 추출**: 정규표현식을 도입하여 제목 내 `25-4차`와 같은 차수 정보 및 `강남구`와 같은 행정구역명을 자동으로 추출.

## 3. 백엔드 검색 및 정렬 엔진 (QueryDSL) 고도화

### 3.1 태그 기반 정밀 쿼리
- **문제 해결**: 제목 텍스트 매칭 방식의 한계를 극복하기 위해 `NOTICE_TAG` 테이블과 조인하여 실제 부여된 태그 값을 기준으로 필터링 수행.
- **타입 격리 필터**: 인스타 필터 클릭 시 `TagType.SYSTEM`인 태그만 타겟팅하도록 명시하여, 이름이 같은 일반 태그와 섞이는 현상 방지.

### 3.2 고차원 정렬 알고리즘
- **알림 키워드 순**: 단순히 제목 포함 여부가 아니라, 사용자의 관심 키워드와 공고 태그가 일치하는 **개수(Count)**를 DB 레벨에서 계산하여 내림차순 정렬.
- **마감 임박 순**: `CaseBuilder`를 사용하여 (오늘 이후 공고 ASC) -> (지난 공고 DESC) 순서로 그룹화 정렬하여 사용자가 놓치지 말아야 할 공고를 최상단 노출.

## 4. 보안 및 데이터 무결성 강화

### 4.1 JWT 보안 강화
- **로그 제거**: 브라우저 개발자 도구(Console)에 Access Token과 Refresh Token의 원문이 노출되던 위험 로그(`axios.js`)를 전면 제거하여 토큰 탈취 위험 차단.

### 4.2 DB 아키텍처 최적화
- **복합 유니크 제약 조건**: `tag` 테이블의 유일성 기준을 `name`에서 `(name, type)`으로 확장하여, 데이터의 다양성(동일 이름/다른 타입)을 보장하면서도 중복 삽입 에러 해결.
- **Enum 예외 처리**: 신규 추가된 `TagType.SYSTEM` 삽입 시 발생하던 `Data truncated` 에러를 방지하기 위한 스키마 확장 가이드 수립.

## 6. [추가] 2026-03-05 심야 긴급 트러블슈팅 및 로직 정교화 리포트

### 6.1 시스템 상태(SYSTEM Tag) 판정 우선순위 조정
- **문제**: "결과 안내"와 같이 키워드가 겹치는 경우 포괄적인 "안내"가 우선 적용되어 "결과발표" 필터에서 누락됨.
- **해결**: `TagService` 및 `NoticeResponseDto`의 상태 결정 로직에서 **`RESULT(결과발표)`의 우선순위를 `INFO(안내)`보다 높게 설정**하여 더 구체적인 정보가 태깅되도록 수정.

### 6.2 정렬 알고리즘 전면 개편 (서브쿼리 방식 도입)
- **문제**: `Left Join`과 `GroupBy`를 이용한 점수 계산 시, 좋아요(UserNotice)와 태그(NoticeTag) 수에 따른 **카테시안 곱(Cartesian Product)** 현상으로 정렬 점수가 뻥튀기되는 오류 발생.
- **해결**: 메인 쿼리에서 조인을 제거하고 **OrderSpecifier 내부에 독립적인 서브쿼리**를 사용하여 `matchScore`(알림 키워드 매칭 수)와 `interestScore`(하트 수)를 격리 계산하도록 전면 재설계. (정확도 및 성능 대폭 향상)

### 6.3 수집기 동시성 충돌 및 트랜잭션 롤백 해결
- **문제**: DB 스키마에 남아있던 레거시 유니크 인덱스(`name`)와 `REQUIRES_NEW` 전파 속성으로 인한 **Lock wait timeout** 에러로 수집이 중단됨.
- **해결**: 
    - 레거시 인덱스(`UK1wdpsed5kna2y38hnbgrnhi5b`) 삭제 가이드 및 실행.
    - `TagService` 내 태그 생성 로직에 **이중 체크(Double-Check) 패턴** 도입하여 중복 에러 발생 시 즉시 재조회 후 반환하도록 보완.
    - 불필요한 트랜잭션 분리를 제거하여 부모 트랜잭션과의 락 경합 해소.

### 6.4 기존 데이터 자동 보정 시스템 (Batch Sync)
- **문제**: 최근 도입된 SYSTEM 태그 로직이 적용되기 전의 기존 데이터(SH 공고 약 1,500건 등)가 필터링에서 누락됨.
- **해결**: `TagService.syncSystemTags()` 메서드 구현 및 `DataInitializer` 연동. 서버 기동 시 **태그가 없는 모든 공고를 스캔하여 자동으로 상태를 판별하고 태그를 채워넣는 보정 로직** 가동.

### 6.5 데이터 정합성 및 워딩 통일
- **알림 키워드 출처 정정**: 정렬 시 사용되지 않는 레거시 `keywords` 테이블 대신 실제 운영 중인 `user_tag` 테이블을 조회하도록 `NoticeController` 수정.
- **워딩 정제**: 프론트엔드/백엔드 전체에서 `정정안내` -> **`안내`**로 명칭 통일 및 `원문확인`, `예정` 등 비표준 상태값 제거 (4대 핵심 상태로 표준화).
- **JPA Auditing 활성화**: `AppApplication`에 `@EnableJpaAuditing` 추가하여 `Tag` 엔티티의 생성/수정 시간이 `null`로 들어오는 문제 해결.
